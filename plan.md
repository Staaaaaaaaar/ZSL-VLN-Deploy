调研结论

ActiveVLN现在主要是训练/评测框架，不是现成的机器狗部署程序。其评测链路是“图像输入 + 文本动作输出 + 离散动作执行”，可作为我们PC推理端的参考实现。核心入口在 eval_vlnce.py 和 parse_utils.py。
动作语义是离散且可控的：前进、左转、右转、停止，且带固定步长（R2R是25cm与15度基元，RxR是25cm与30度基元），这非常适合做“保守映射”到机器狗速度命令。参考 env.py 和 constants.py。
zsibot_sdk提供了我们需要的高层控制与状态读取接口，含移动、站立、趴下、IMU、速度、电量等；并且有本体RTSP视频流文档，和你的“视频流或图片输入”策略完全匹配。参考 api_zsl-1.md, api_zsl-1w.md, video.md。
当前工作区只有三方代码，没有你的主部署工程目录。参考 third_party。
保守部署原则

先只用HighLevel接口，不碰LowLevel，避免控制风险与切换冲突。依据 faq.md。
先实现单机单狗单线程闭环：取帧 -> 推理 -> 一条安全动作 -> 执行 -> 反馈。
动作先限制为三种：前进、原地左转、原地右转、停止；禁用跳跃、特技。
先做低频推理+限速执行，保证“可运行+可停机+可恢复”优先。
建议的模块化方案（与你的设想对齐）

core：统一数据类型与协议
robot：机器狗连接、状态获取、命令发布
model：ActiveVLN推理客户端（OpenAI兼容接口）
runtime：主控制循环与状态机
converters：模型动作到机器人速度命令的统一转换
io：视频流与图片输入适配
safety：急停、超时、心跳、限速、异常降级
分阶段计划（每步都可验收）

M0 环境与烟测
产物：机器人连接脚本、RTSP取流脚本、模型服务连通脚本。
验收：三条链路分别独立通过。
M1 core与配置
产物：统一类型定义、动作枚举、配置文件。
验收：类型检查通过，配置可驱动后续模块。
M2 robot适配层
产物：ZsiBotHighLevelAdapter，封装init、check、move、stand、lie、state。
验收：可完成站起-前进-停止-趴下闭环。
M3 model适配层
产物：ActiveVLNClient，输入单帧+指令，输出离散动作序列。
验收：对固定图片集返回可解析动作。
M4 converters与safety
产物：动作映射器、限幅器、超时处理、非法输出兜底停止。
验收：随机脏输出不会导致危险动作。
M5 runtime总控
产物：单进程主循环（视频流/图片两种输入），支持开始、暂停、急停。
验收：实机连续运行10分钟无失控。
M6 基础测试与回归
产物：最小单元测试+集成测试脚本。
验收：关键路径一键复现。
